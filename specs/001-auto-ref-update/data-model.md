# Data Model: Automated Reference Material Update System

**Branch**: `001-auto-ref-update` | **Date**: 2026-03-09

---

## Entities

### 1. `sources.json` — Pipeline Configuration

Already fully specified in `context/sources-json-schema.md`. Reproduced here for traceability.

**Location**: `sources.json` (repo root)

**Shape**:
```
{
  version: string           // Schema version, bump on breaking changes
  repos: RepoEntry[]
}

RepoEntry {
  id: string                // Unique kebab-case identifier
  github_url: string        // Full HTTPS URL, no trailing slash
  sparse_paths: string[]    // Gitignore-style paths; dirs end with /
  ref: "latest-release" | "main" | "vX.Y.Z"
  trigger: "release" | "schedule" | "both"
  updates: UpdateEntry[]    // Must be non-empty
}

UpdateEntry {
  reference_file: string    // Must start with plugins/
  section?: string          // Required when multiple repos target same file
  generator_agent: "claude" | "codex" | "gemini" | "copilot"
  reviewer_agent: "claude" | "codex" | "gemini" | "copilot"
                            // Must differ from generator_agent
}
```

**Validation rules** (enforced at workflow start, per schema doc):
1. All `id` values unique
2. All `github_url` values reachable
3. All `reference_file` paths start with `plugins/`
4. `generator_agent !== reviewer_agent` per `UpdateEntry`
5. Multiple `UpdateEntry` targeting same file must all declare `section`
6. No `sparse_paths` entry includes `node_modules/`, `dist/`, `target/`, `.git/`

### 2. Baseline Git Tags — Per-Repo Processing Record

Persists the last-successfully-processed state per source repo as a force-updated annotated git tag in the plugin repo. No file commit required.

**Tag naming**: `refs/tags/ref-update/<repo-id>`

**Tag message format** (single line, pipe-delimited):
```
<ref> | <commit-sha> | <processed_at-ISO8601>
```

**Examples**:
```
tag: refs/tags/ref-update/fastedge-sdk-js
msg: v2.1.0 | abc1234def5678abc1234def5678abc1234def56 | 2026-03-09T14:30:00Z

tag: refs/tags/ref-update/fastedge-sdk-rust
msg: v0.4.1 | 999aaa111bbb222ccc333ddd444eee555fff6666 | 2026-03-09T14:32:00Z
```

**State transitions**:
- First run (no tag exists): treat as new content, run full pipeline; create annotated tag on success
- Subsequent runs: fetch tag message → parse commit SHA → compare to current source HEAD → skip if identical
- After successful PR open: force-push tag with updated commit SHA, ref, and processed_at

**Tag push command**:
```bash
git tag -f -a "ref-update/$REPO_ID" \
  -m "$REF | $COMMIT | $(date -u +%Y-%m-%dT%H:%M:%SZ)"
git push origin --force "refs/tags/ref-update/$REPO_ID"
```

**Tag read command**:
```bash
# Get the annotated tag object SHA, then read the message
TAG_MSG=$(git ls-remote origin "refs/tags/ref-update/$REPO_ID" \
  | awk '{print $1}' \
  | xargs -I{} git cat-file tag {} 2>/dev/null \
  | grep -A1 "^$" | tail -1)
LAST_COMMIT=$(echo "$TAG_MSG" | cut -d'|' -f2 | tr -d ' ')
```

**Why not a committed file**: A JSON file would require a commit to some branch. `main` is protected; the `auto-ref-update/<repo-id>` PR branch would pollute the PR diff with unrelated baseline commits. Tags are branchless, atomic on push, and queryable via `git ls-remote` without a checkout.

### 3. Traceability Frontmatter — Reference File Header

Prepended/updated at the top of every reference file touched by automation. Format is already specified in `context/sources-json-schema.md`.

**Shape** (embedded as HTML comment at top of Markdown file):
```
<!--
  auto-updated: true
  sources:
    - id: string            // matches sources.json repo id
      ref: string           // tag or branch that was checked out
      commit: string        // full commit SHA
      updated: string       // ISO date YYYY-MM-DD
-->
```

**Rules**:
- Must be the very first content in the file
- Review agent validates presence and correctness before approving
- A missing or malformed block is a review blocker (Principle XV)
- When multiple repos update sections of the same file, all contributing repos appear under `sources:`

### 4. Pull Request — One Per Changed Source Repo

**Branch naming**: `auto-ref-update/<repo-id>` (stable name, same branch reused on re-runs so existing PR gets updated)

**PR shape**:
```
title:  "auto: update reference docs from <repo-id> (<ref>)"
labels: ["auto-ref-update"]
        + ["needs-review"]   // added when reviewer verdict is "reject"
body:
  ## Source
  - Repo: <github_url>
  - Ref: <ref>
  - Commit: <commit>

  ## Changes
  <list of reference files updated, with section if applicable>

  ## Review Agent Findings
  <reviewer_agent> verdict: <ACCEPT | REJECT>

  <verbatim review output>

  ---
  _Generated by sync-reference-docs workflow. Do not edit this PR body manually._
```

**Lifecycle**:
- Opened on first run where changes detected for that repo
- Body replaced (not appended) on subsequent runs that update the same open PR
- Review findings always posted as fresh body update (not a separate comment)
- Humans merge; automation never merges

### 5. Workflow Run Summary — Pipeline Outcome Log

Not persisted to a file. Written to GitHub Actions step summary (`$GITHUB_STEP_SUMMARY`).

**Shape** (Markdown table in Actions summary):
```
| Source Repo       | Outcome  | Ref     | PR                      |
|-------------------|----------|---------|-------------------------|
| fastedge-sdk-js   | ✅ PR #42 | v2.1.0  | https://github.com/...  |
| fastedge-sdk-rust | ⏭ skipped | v0.4.1 | no changes              |
| fastedge-test     | ❌ failed | main    | fetch error             |
```

## Entity Relationships

```
sources.json
  └── RepoEntry [1..*]
        ├── triggers sparse checkout
        ├── drives UpdateEntry [1..*]
        │     ├── maps to reference_file (in plugins/)
        │     ├── specifies generator_agent
        │     └── specifies reviewer_agent
        └── has baseline in annotated git tag refs/tags/ref-update/<id>

Pipeline Run
  └── processes each RepoEntry
        ├── reads baseline → decides skip/proceed
        ├── runs generator_agent → produces staged content
        ├── runs reviewer_agent → produces verdict + findings
        ├── if changed + accepted: writes plugins/, updates frontmatter
        └── opens/updates Pull Request
              └── body contains review findings
```
